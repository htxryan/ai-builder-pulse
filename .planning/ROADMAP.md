# Roadmap: AI Builder Pulse

## Overview

Build a fully automated daily newsletter pipeline: a GitHub Actions cron job collects content from HN, GitHub Trending, Reddit, and RSS feeds, passes candidates through Claude for relevance scoring and categorization, verifies every link traces to source data, and publishes the formatted issue to Buttondown — committing each daily output to the repo as a verifiable artifact. No human in the loop.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Foundation** - GitHub Actions scaffold, secrets config, keepalive, and error alerting
- [ ] **Phase 2: Content Ingestion** - Multi-source collection (HN, GitHub Trending, Reddit, RSS) with freshness enforcement and deduplication
- [ ] **Phase 3: AI Curation & Link Integrity** - Claude filtering, categorization, description generation, cost logging, and URL provenance validation
- [ ] **Phase 4: Publishing** - Buttondown API integration with idempotency, empty-issue guard, and delivery verification
- [ ] **Phase 5: Persistence** - Daily issue committed to git as markdown and JSON for archiving and future weekly digest

## Phase Details

### Phase 1: Foundation
**Goal**: The automated pipeline infrastructure exists and can be triggered on a schedule without human intervention
**Depends on**: Nothing (first phase)
**Requirements**: AUTO-01, AUTO-02, AUTO-03, AUTO-04
**Success Criteria** (what must be TRUE):
  1. A GitHub Actions workflow runs on a daily cron schedule and completes successfully
  2. API keys (Claude, Buttondown, Reddit) are stored as GitHub Actions secrets and accessed by the workflow without being exposed in logs
  3. A keepalive mechanism prevents the workflow from being auto-disabled after 60 days of inactivity
  4. When the pipeline fails, an error notification is triggered via GitHub Actions (email or annotation)

### Phase 2: Content Ingestion
**Goal**: The pipeline collects fresh, deduplicated content from all four sources and delivers a normalized candidate list
**Depends on**: Phase 1
**Requirements**: INGEST-01, INGEST-02, INGEST-03, INGEST-04, INGEST-05, INGEST-06
**Success Criteria** (what must be TRUE):
  1. HN stories from the last 24 hours (300-500 items) are fetched via the Algolia API and available as structured data
  2. GitHub Trending AI repositories are scraped and available as structured items
  3. Posts from Reddit AI communities (r/LocalLLaMA, r/MachineLearning) from the last 24 hours are collected via PRAW
  4. New posts from key AI blog RSS/Atom feeds are parsed and available as structured items
  5. All collected items older than 24 hours are excluded before passing downstream
  6. Items sharing the same normalized URL across sources appear only once in the candidate list
**Plans**: TBD

### Phase 3: AI Curation & Link Integrity
**Goal**: The candidate list is filtered for builder relevance, categorized, described, and every output URL is proven to come from source data — not Claude
**Depends on**: Phase 2
**Requirements**: CURATE-01, CURATE-02, CURATE-03, CURATE-04, CURATE-05, LINK-01, LINK-02, LINK-03
**Success Criteria** (what must be TRUE):
  1. Claude receives pre-filtered candidates (keyword/score threshold removes 60-80% before the API call) and returns structured JSON with relevance scores
  2. Each accepted item is assigned to exactly one of the fixed taxonomy categories (Tools & Launches, Model Releases, Techniques & Patterns, Infrastructure & Deployment, Notable Discussions, Think Pieces & Analysis, News in Brief)
  3. Each accepted item has a 1-2 sentence description generated by Claude
  4. Every URL in the formatted output can be traced verbatim to a field in the raw ingested data — no Claude-generated URLs pass through
  5. A post-generation validation step rejects any output URL not present in the input data set
  6. Token counts and estimated API cost for each run are logged
**Plans**: TBD

### Phase 4: Publishing
**Goal**: The curated, formatted issue is reliably delivered to Buttondown subscribers with guards against empty sends and duplicate delivery
**Depends on**: Phase 3
**Requirements**: PUB-01, PUB-02, PUB-03, PUB-04
**Success Criteria** (what must be TRUE):
  1. A formatted markdown issue is published to Buttondown via the REST API and appears as a sent newsletter
  2. Re-running the pipeline for the same date does not publish a duplicate issue
  3. After publishing, the pipeline confirms successful delivery via Buttondown API response status
  4. When the curated item count falls below the minimum threshold, the pipeline skips publishing and logs the skip reason
**Plans**: TBD

### Phase 5: Persistence
**Goal**: Each day's issue is committed to the git repo as structured files, creating a durable archive that enables the future weekly digest
**Depends on**: Phase 4
**Requirements**: PERSIST-01, PERSIST-02
**Success Criteria** (what must be TRUE):
  1. After each successful pipeline run, the daily issue is committed to the repo as both a markdown file and a JSON file
  2. Files are written under a consistent directory structure (issues/YYYY-MM-DD/) that any future rollup script can reliably read
