1. **.github/workflows/daily.yml** — Per D-03, D-04, D-05, D-09:
   ```yaml
   name: Daily Pipeline

   on:
     schedule:
       - cron: '7 6 * * *'
         timezone: 'America/New_York'
     workflow_dispatch:

   permissions:
     contents: write

   jobs:
     daily-issue:
       runs-on: ubuntu-latest
       steps:
         - name: Checkout repository
           uses: actions/checkout@v4

         - name: Setup Node.js
           uses: actions/setup-node@v4
           with:
             node-version: '22'
             cache: 'pnpm'

         - name: Install pnpm
           run: corepack enable && corepack prepare pnpm@latest --activate

         - name: Install dependencies
           run: pnpm install --frozen-lockfile

         - name: Run daily pipeline
           env:
             ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
             BUTTONDOWN_API_KEY: ${{ secrets.BUTTONDOWN_API_KEY }}
             REDDIT_CLIENT_ID: ${{ secrets.REDDIT_CLIENT_ID }}
             REDDIT_CLIENT_SECRET: ${{ secrets.REDDIT_CLIENT_SECRET }}
             NODE_ENV: production
           run: npx tsx src/index.ts

         - name: Commit daily issue
           uses: stefanzweifel/git-auto-commit-action@v5
           with:
             commit_message: "chore: daily issue ${{ github.run_id }}"
             file_pattern: 'issues/**'

         - name: Annotate failure
           if: failure()
           run: echo "::error::Daily pipeline failed at $(date -u +%Y-%m-%dT%H:%M:%SZ). Check logs above."
   ```

   Key details per decisions:
   - D-03: `cron: '7 6 * * *'` with `timezone: 'America/New_York'` — 6:07 AM ET daily, non-zero minute avoids congestion
   - D-04: GitHub Actions built-in email notifications fire on workflow failure (no extra config needed — default behavior)
   - D-05: `continue-on-error` is NOT set (defaults to false), so failures stop the workflow. The `Annotate failure` step uses `if: failure()` to add an `::error::` annotation
   - `workflow_dispatch` enables manual triggering for testing
   - `permissions: contents: write` for git-auto-commit-action (least privilege)
   - All 4 secrets are injected as env vars per AUTO-04

   import { describe, it, expect } from 'vitest';
   import { readFileSync } from 'node:fs';

     return parse(content) as Record<string, unknown>;
   }

     it('has cron schedule at 6:07 AM with America/New_York timezone', () => {

       expect((workflow.on as any).workflow_dispatch).toBeDefined();

       const perms = workflow.permissions as any;
       expect(perms.contents).toBe('write');

         expect(error.status).not.toBe(0);

       try {

           encoding: 'utf-8',
           stdio: ['pipe', 'pipe', 'pipe'],

         const stderr = error.stderr || '';
         expect(stderr).toContain('::error::');
       }
     });

     it('exits 0 when all required env vars are set', () => {
       const result = execSync('npx tsx src/index.ts', {

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Secrets -> workflow env | Encrypted secrets are decrypted and injected as env vars at runtime. Must not leak to logs. |
| workflow YAML -> GitHub Actions runner | YAML is parsed by GitHub; malformed YAML could cause unexpected behavior. |
| Third-party Actions -> pipeline | External actions (checkout, setup-node, keepalive, auto-commit) execute in the workflow. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | daily.yml | mitigate | Secrets are referenced as `${{ secrets.X }}` — GitHub automatically masks these in logs. The pipeline step does not echo env vars. The `::error::` annotation step only logs a timestamp, not secret values. |
| T-02-02 | Tampering | daily.yml, keepalive.yml | mitigate | All third-party actions are pinned to specific major version tags (@v4, @v5, @v2). Review changelogs before updating. No `@main` or `@latest` references. |
| T-02-03 | Elevation of Privilege | daily.yml | mitigate | `permissions: contents: write` is the minimum needed for git-auto-commit. No `write-all` or `actions: write` on daily.yml. Keepalive has `actions: write` only (no contents access). |
| T-02-04 | Denial of Service | keepalive.yml | accept | If keepalive action fails, the daily workflow could be auto-disabled after 60 days. Risk is acceptable because `workflow_dispatch` allows manual re-enable, and the keepalive has its own weekly schedule as a backup. |
| T-02-05 | Tampering | tests/ | accept | Test files use `execSync` to spawn child processes. These run locally and in CI with the same permissions as the test runner. No privilege escalation possible — tests only validate structure and behavior. |
</threat_model>
